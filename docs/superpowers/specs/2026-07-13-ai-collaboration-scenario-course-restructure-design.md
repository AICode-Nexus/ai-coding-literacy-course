# 《AI 协同方法论》场景化课程重构设计规格

状态：已由用户于 2026-07-13 逐段确认并复核批准。

本规格替代 `2026-07-13-vitepress-dual-mode-course-design.md` 中的课程目标解释、章节结构、案例策略、教材目录与内容验收要求。原规格中的 VitePress 双模式架构、16:9 舞台约束、离线能力和基础交互要求继续有效；如两份规格冲突，以本规格为准。

## 1. 课程定位

课程名称：**AI 协同方法论：从“会用 AI”到“善用 AI”**。

课程目标以官方课程说明为最高约束：讲解 AI 思维，帮助校招生掌握“怎么用 AI 提效、用什么思路用 AI”，形成“人机协作”的工作思维。

课程不以记忆工具功能、模型型号或 Agent 术语为终点。课程结束后，学员应能解释并实践以下工作方法：

1. 把模糊要求转成目标明确、材料清楚、结果可验收的任务。
2. 判断哪些工作适合交给 AI，哪些判断、授权与责任必须由人承担。
3. 根据任务选择模型、工具、Agent 和编排方式，而不是追逐工具清单。
4. 用证据、评测、边界和人工检查点控制交付质量。
5. 把一次提效沉淀为模板、检查清单、知识资产、Skill 或可持续工作流程。

授课配置：

- 80 分钟讲授，15 分钟交流。
- 约 1050 人的大课。
- 讲课为主，课堂展示需要丰富、清晰、可在远距离阅读。
- 舞台按 16:9、1920×1080 逻辑画布设计并支持浏览器全屏。
- 课程网站同时承担讲师展示与学员课后自学。
- 不与 Kooky 课程或 AI 大赛建立叙事衔接。

## 2. 学员依据与教学判断

《AI Coding 素养课程参考材料》提供的可读统计为：

- 超过九成学员使用过通用大模型工具。
- 78.21% 使用过 AI Coding 类工具。
- 21.23% 使用过智能体或工作流工具。
- 25.70% 接触过 AI 图像或视频工具。

附件中的“26 届校招生基本情况”没有可读取的岗位明细，因此课程不得假定全体学员都是研发岗。

据此采用以下教学判断：

- 不从“第一次使用 AI”讲起。
- 不把 Coding Agent 当作所有人的默认入口。
- 重点补齐从单点问答、内容生成到多步骤协同与流程复用的认知缺口。
- 主线必须跨岗位可理解；研发、产品、运营、职能通过迁移示例获得各自入口。
- A 类学员掌握核心协同方法，少量 B 类学员可继续进入 Agent、Harness、Evals 等进阶内容。

## 3. 核心设计原则

### 3.1 先给地图，再走路线

概念、工具与知识点必须在课堂材料中显性展示，不能全部折叠到附录。节点 03 先建立 AI 基础、模型、工具、Agent 工作系统和质量控制的全景地图，只讲位置与关系；后续场景再逐一解释它们何时有用。

### 3.2 场景驱动，不编造真实案例

课程不再设置虚构的“真实案例剧场”，也不以一条具体业务故事贯穿全程。每个节点使用一个跨岗位可代入的场景命题，例如“收到一句你先弄一下，第一步是什么”。

场景只提供：

- 普遍困境；
- 普通做法与协同做法；
- 对应概念和工具；
- 决策框架、检查点与可带走结论；
- 可由讲师替换的岗位或项目入口。

场景不得包含虚构客户、虚构内部项目、编造提效数字或无法验证的业务结果。只有拥有可追溯材料的内容才能标为“真实案例”；否则统一标为“场景演示”或“应用示例”。

### 3.3 知识至少出现两次

每个核心概念至少出现两次：

1. 在全景地图中出现，帮助学员定位它与其他概念的关系。
2. 在场景中再次出现，说明它解决什么、不能替代什么、何时使用。

### 3.4 专业判断在前，AI 能力横向放大

课程第一章必须是“AI 时代的 T 型人才”。纵向是岗位知识、业务理解、质量判断和责任承担；横向是检索、分析、表达、执行、协作、验收与复盘等 AI 协同能力。

T 型人才不是一个人替代所有岗位，而是理解相邻岗位的输入与输出，并借助 AI 完成一个可控的小闭环。

## 4. 九个课程节点与 80 分钟结构

| 时间 | 导航节点 | 课堂场景命题 | 主要内容 |
| --- | --- | --- | --- |
| 0–8 分钟 | 01 AI 时代的 T 型人才 | AI 时代，只做好本职工作就够了吗？ | 一专多能、人机协作、相邻岗位小闭环 |
| 8–12 分钟 | 02 从工具使用到人机协同 | 大家都会用 AI，为什么结果差距仍然很大？ | 学员统计、问答式使用、任务式协同、工作系统 |
| 12–24 分钟 | 03 AI 基础、模型与工具全景 | 面对一项任务，应该选择哪个模型和工具？ | LLM 基础、模型选择、工具分类，以及 Prompt、Context、Tool、Agent、MCP、Skill、Harness、Eval、Guardrails 的全景定位 |
| 24–34 分钟 | 04 AI 协同七步法 | 收到一句“你先弄一下”，第一步是什么？ | Goal、Context、Task、Output、Eval、Guardrails、Loop |
| 34–43 分钟 | 05 Context：组织正确的信息 | 材料越多，AI 为什么反而越容易混乱？ | Context Engineering、优先级、检索、知识库、持久上下文 |
| 43–55 分钟 | 06 Agent：从回答走向行动 | 一个聊天框做不完，要不要使用多个 Agent？ | Tool、MCP、Agent、Skill、Subagent、Team、Worktree、Harness |
| 55–66 分钟 | 07 质量、安全与人的责任 | AI 已经给出结果，你敢直接交付吗？ | Eval、Dataset、Grader、Tracing、Guardrails、审批与安全 |
| 66–74 分钟 | 08 从一次提效到流程重构 | 这次做完了，下次还要从头再来吗？ | Loop、模板、检查清单、知识沉淀、Skill、工作流重构 |
| 74–80 分钟 | 09 岗位迁移与行动计划 | 回到自己的岗位，明天可以改造哪个小流程？ | 研发、产品、运营、职能迁移，七步法回顾与行动 |
| 80–95 分钟 | 交流与深挖 | 现场最想继续下钻哪一层？ | 工具选型、概念边界、场景讲解、任务卡演示 |

九个讲授节点合计必须等于 80 分钟。节点名称同时用于讲师总览和教材侧栏，课堂大标题可使用对应的场景问句。

## 5. 场景展开语法

每个节点从下列场景组件中选择 3–5 个，不要求每个节点机械重复全部组件：

1. **场景问题**：一句跨岗位问题，引发全场判断。
2. **现场选择**：举手、口头选择或独立思考，不依赖 1050 人同时联网。
3. **普通做法 / 协同做法**：展示行为差异，而不是只展示结果差异。
4. **知识地图**：在本节点点亮相关概念与工具。
5. **决策框架**：说明何时选模型、工具、Agent 或人工处理。
6. **边界卡**：说明概念相邻关系与不能替代的能力。
7. **检查点**：展示证据、权限、验收和停止条件。
8. **带走结论**：一句可迁移的方法，连接下一节点。

讲师备注包含：

- 本屏教学目的；
- 可代入的研发、产品、运营或职能方向；
- 可选追问；
- 容易讲错的边界；
- 可跳过或深入的 B 层内容。

备注不得写成必须逐字朗读的虚构故事脚本。

## 6. 课堂显性知识体系

### 6.1 AI 基础与模型认知

- LLM、Token、Context Window、Reasoning / Thinking、多模态、幻觉。
- 成本不只看单价，还看延迟、一次成功率、返工、上下文与风险。
- 模型选择使用六个维度：任务难度、上下文规模、多模态需求、工具能力、数据安全、总成本。
- 不在主教材写死“最强模型”、具体跑分、长期不变的窗口和未来型号。

### 6.2 任务表达与协同方法

- Goal、Context、Prompt、Spec、Task、Output、Eval、Guardrails、Loop。
- 任务卡至少包含目标、材料、动作、交付物、验收与边界。
- Prompt 是任务说明的一部分，不是独立于上下文、工具和验收的咒语。

### 6.3 行动与 Agent 系统

- Tool Use / Function Calling、Agent、MCP、Skill。
- Single Agent、Subagent、Agent Team、handoff、manager-as-tool、Worktree。
- Agent Loop：Goal → Plan → Act → Observe → Verify → Continue / Stop。
- MCP 是连接与上下文交换协议，核心包括 tools、resources、prompts；它不是 Agent，也不负责业务编排。
- 默认选择能完成任务的最简单结构；只有独立性、契约、权限或通信方式变化时才增加 Agent。

### 6.4 质量、安全与责任

- Eval、Dataset、Grader、Tracing、Guardrails、Human-in-the-loop。
- 确定性检查、AI 语义检查、责任人确认三层验收。
- Prompt Injection、最小权限、Sandbox、审批、停止条件、预算和回滚。
- AI 适合搜索、生成、比较、执行和预检；人负责目标、取舍、授权、承诺和后果。

### 6.5 工作系统与知识复用

- Context Engineering、Harness Engineering、Loop、持久状态。
- RAG、项目知识库、AI 维护 Wiki、模板、检查清单、Skill。
- LLMWiki、Loop Engineering 等非稳定行业标准名称只能标为前沿观察或实现思路，不能包装为成熟共识。
- 原大纲中的“Claude Tag”没有稳定通用定义，替换为“项目规则与持久指令”，可用 AGENTS.md、CLAUDE.md 等具体例子说明。

### 6.6 成熟度标签

- **核心必学**：Goal、Context、Prompt、Output、Eval、Guardrails、Loop、Tool、Agent。
- **进阶掌握**：MCP、Skill、Subagent、Worktree、Tracing、Harness。
- **前沿了解**：Agent Team、AI 维护 Wiki、Loop Engineering 等仍在快速演进的方式或标签。

前沿标签必须解释成熟度、适用边界和协调成本；不得暗示“越复杂越先进”。

## 7. 工具与模型全景

课堂展示工具类别与选择条件，代表产品只作为示例，不作为排名：

| 类别 | 可展示的代表 | 主要判断 |
| --- | --- | --- |
| 通用 AI 助手 | 星火、ChatGPT、Claude、Gemini、豆包、Kimi、通义千问、DeepSeek | 推理、上下文、多模态、隐私、成本 |
| 深度研究 | 各类 Deep Research、联网检索与资料分析工具 | 来源质量、引用可追溯、时效性 |
| Coding Agent | Codex、Claude Code、Cursor、GitHub Copilot | 仓库理解、工具调用、测试、权限、回滚 |
| Agent / 工作流平台 | Dify、Coze、n8n 等 | 流程编排、系统连接、审批、日志、失败恢复 |
| 多模态创作 | 即梦、可灵、Midjourney 等 | 图像、视频、可控性、版权边界 |
| 知识与连接 | RAG、知识库、MCP、企业数据连接器 | 权威来源、权限、更新、可追溯 |
| 能力复用 | Prompt 模板、规则文件、Skill、检查清单 | 稳定复用、版本化、可评测性 |

所有时效性强的工具条目都要保存官方入口和 `verifiedAt`。过期条目可继续出现在历史记录中，但不得继续显示为当前推荐。

## 8. 双模式课程网站

### 8.1 首页

首页提供两个同等清晰的入口：

- **开始讲课**：进入 `/present`。
- **课后学习**：进入课程地图或第一章。

首页展示课程目标、9 个节点、80 分钟路线、学习产出和内容规模，不用营销式夸张数字。

### 8.2 讲师模式

- 9 个节点，预计 30–34 个全屏场景。
- 1920×1080 逻辑画布按视口等比缩放并居中。
- 支持方向键、空格、PageUp/PageDown、Home/End、F 全屏、O 总览、Esc 退出。
- URL 保存当前场景；无效场景参数回退第一页。
- 主内容置于 8% 安全区；最小字号不低于 28px。
- 不依赖公网字体、图片、API 或现场网络。
- 需要演示时，同时提供目标、输入、预期产物和静态兜底结果。

讲师场景类型继续包括 Thesis、Compare、Flow、Architecture、Audience、Demo、Checklist；原 `Case Theatre` 改成 `Scenario`，不再暗示虚构内容是真实案例。

### 8.3 教材模式

教材与讲师模式使用完全相同的 9 个节点：

- `/guide/00-start`：课程地图、目标、受众、学习方式。
- `/guide/01-t-shaped`：AI 时代的 T 型人才。
- `/guide/02-collaboration-shift`：从工具使用到人机协同。
- `/guide/03-ai-landscape`：AI 基础、模型与工具全景。
- `/guide/04-method`：AI 协同七步法与任务卡。
- `/guide/05-context`：Context Engineering、检索、知识与持久状态。
- `/guide/06-agent`：Agent Loop、Tool、MCP、Skill 与编排边界。
- `/guide/07-quality`：Evals、Tracing、安全、审批与人的责任。
- `/guide/08-workflow`：Harness、Loop、知识沉淀与流程重构。
- `/guide/09-transfer`：岗位迁移、7 天行动和结业产物。

每个教程章节包含：

1. 本章结论；
2. 通用工作场景；
3. 概念关系图；
4. 详细知识解释；
5. 工具与选择方法；
6. 常见误区和边界；
7. 岗位迁移示例；
8. 一个练习；
9. 官方来源与延伸阅读。

### 8.4 附录

- `/appendix/glossary`：概念词典、成熟度和关联节点。
- `/appendix/tool-radar`：工具类别、选择维度、官方入口和核验时间。
- `/appendix/templates`：任务卡、材料清单、验收清单、风险清单、复盘模板。
- `/sources`：公开来源、内部材料说明和更新记录。

## 9. 单一内容源与边界

讲师模式、教材、词典和工具雷达共享同一套事实数据：

- `course.js`：课程元信息、节点、时间和产出。
- `concepts.js`：定义、边界、成熟度、相关场景和来源。
- `tools.js`：类别、代表、选择条件、官方入口和核验日期。
- `scenarios.js`：通用场景命题、可代入方向和教学框架。
- `scenes.js`：讲师屏幕，只引用共享概念、工具和场景 ID。
- `sources.js`：公开与内部来源、时效性和核验说明。
- `templates.js`：任务卡、检查清单和练习模板。

现有 `cases.js` 中能够追溯来源的内部内容可以迁移到来源页或“经核验案例”附录；其余内容改名为 `examples.js` 或迁入场景的岗位示例，不能继续统一显示为真实案例。

## 10. 失败与降级策略

- 工具或模型信息超过维护周期时，显示“待重新核验”，不继续显示当前推荐状态。
- 官方链接失效不阻断课程构建，但链接检查必须报告问题。
- 讲师模式加载不到可选增强素材时，使用内置 CSS 图解和文本结论。
- 任何现场演示失败时，讲师可立即切到静态过程与预期结果。
- 缺少内部案例来源时，内容必须降级为场景或示例，而不是补写业务细节。
- 教材交互失效时，核心定义、方法和示例仍以静态 HTML 可读。

## 11. 验收与测试

### 11.1 内容契约

- 9 个讲授节点时间合计等于 80 分钟。
- 每个节点同时存在于课程数据、讲师总览和教材侧栏。
- 每个核心概念至少在全景图出现一次，并在一个场景中再次出现。
- 概念卡包含定义、解决的问题、不能替代的能力、关联场景和成熟度。
- 工具条目包含类别、选择条件、官方入口和核验日期。
- 所有场景标记为“场景演示”或“应用示例”；只有有来源的内容可标为真实案例。
- 不出现虚构客户、项目、数据、提效结果或未经核验的模型排名。

### 11.2 讲师模式

- 场景总数为 30–34。
- 1920×1080 下无滚动条、内容溢出和安全区越界。
- 全屏、键盘、总览、URL 恢复和错误参数回退可用。
- 所有 9 个节点均可从总览直接跳转。
- 现场构建产物断网可运行。

### 11.3 教材模式

- 9 个章节均包含结论、场景、知识、边界、迁移、练习和来源。
- 侧栏、目录、全文搜索、上一章/下一章和移动端导航可用。
- 375、768、1440 和 1920 宽度下无横向溢出。
- 交互组件可键盘操作，并提供清晰焦点状态。

### 11.4 工程验证

- Node 22 环境下 `npm ci`、`npm test`、`npm run build` 通过。
- GitHub Pages base 与本地 `/` base 均通过构建验证。
- 关键讲师场景进行 1920×1080 截图核验。
- 站内链接与官方外链进行自动检查；外链失效作为报告项，不导致核心教材不可读。

## 12. 非目标

- 不开发独立 LMS、登录、学习进度云同步或后台管理系统。
- 不把课程改成 Kooky、某个模型或某个厂商工具的操作培训。
- 不维护会快速失真的模型跑分榜、价格榜或“当前最强”排名。
- 不要求 1050 名学员同时在线互动。
- 不把所有进阶概念讲到 API 或配置级别；课堂讲清边界，教材提供延伸入口。
- 不为了视觉丰富而引入现场必须联网的素材或动画。
