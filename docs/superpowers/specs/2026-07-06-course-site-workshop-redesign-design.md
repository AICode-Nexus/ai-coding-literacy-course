# AI Coding 素养课工作坊型课程站改版设计

## 背景与问题

当前站点已经具备静态课程页、术语筛选、案例切换、任务卡生成、复制、全屏展示等基础能力，但整体仍像“资料展示页 + 零散工具”。用户反馈集中在五类问题：

- 页面布局不稳，局部存在挤压、溢出、空白失衡和视觉重心偏移。
- 部分按钮和控件看起来可交互，但没有真实行为或行为弱，例如首页术语库的圆点和左右翻页按钮。
- 控件样式、按钮层级、交互反馈不够一致。
- 内容密度不足，无法扎实支撑一堂 90 分钟课程。
- 行政审批类案例与 AI Coding 素养课、校招生 AI 协同工作定位不一致。

本次改版按“90 分钟工作坊型课程站”处理，而不是局部 CSS 修补。站点要能直接支持授课、现场练习、案例复盘和课后训练。

## 目标

- 将页面主线改成一堂可讲、可练、可带走模板的 90 分钟工作坊。
- 删除行政审批相关内容，替换为校招生和互联网/AI 工作场景更贴近的岗位案例。
- 将内容组织到固定协同公式：Goal -> Context -> Prompt -> Output -> Eval -> Guardrails -> Loop。
- 增强任务卡生成器，使其支持多类模板、完整任务卡输出、复制反馈和课堂练习。
- 将 7 天行动从单薄清单升级为训练计划，每天包含目标、任务、产出物、检查项和反馈问题。
- 修复无效按钮、控件不一致、布局挤压、文字溢出和首屏信息过载问题。
- 保持静态站架构，不引入前端框架或后端依赖。

## 非目标

- 不重命名课程主品牌，仍保留“AI Coding 素养课：从会用工具到 AI 协同工作”。
- 不引入登录、学习进度存储、后端 API、用户账号或在线批改。
- 不把页面改成知识库型长文站；案例和术语服务课堂路径，而不是无限扩展。
- 不复用行政审批、制度审批、回购审批等职能审批类案例。
- 不增加与课程目标无关的营销型 CTA。

## 信息架构与 90 分钟课堂节奏

导航保留 5 个主入口，但命名贴近课堂：

- 课程总览
- 90分钟路径
- 方法工具箱
- 岗位实战
- 课后训练

页面按真实授课顺序组织：

1. 开场 8 分钟：为什么不是“会用工具”
   - 讲清 T 型人才：一专更深，多能成链。
   - 首屏展示课程目标、90 分钟安排和最终产出。
   - 首屏只保留一个主要行动：开始 90 分钟路径。

2. 模块一 15 分钟：AI 协同工作法
   - 用 Goal、Context、Prompt、Output、Eval、Guardrails、Loop 串成工作链。
   - 避免术语堆叠，让每个概念绑定工作痛点和新人用法。

3. 模块二 20 分钟：从问题到任务卡
   - 现场演示把“帮我做个方案/周报/需求分析”改写成可交给 AI 的任务卡。
   - 任务卡生成器提供模板切换、可编辑输入、结构化输出和复制。

4. 模块三 25 分钟：岗位实战案例
   - 使用校招生高频任务案例。
   - 每个案例包含旧做法、AI 协同做法、人工验收点、可复制提示词、常见坑和验收清单。

5. 模块四 12 分钟：质量验收与边界
   - 专门讲事实核验、责任边界、敏感信息、输出质量标准。
   - 防止“AI 写了就交”的错误使用方式。

6. 收束 10 分钟：7 天训练计划
   - 每天都有任务、产出物、检查项和复盘问题。
   - 最终形成新人 AI 协同工作包：任务卡、提示词、验收清单、复盘记录。

## 内容体系

全站内容围绕一个固定公式：

```text
目标 Goal -> 材料 Context -> 任务 Prompt -> 产出 Output -> 验收 Eval -> 边界 Guardrails -> 复盘 Loop
```

### T 型人才模块

不再只呈现一个巨大 T 字图形。改成三层能力模型：

- 本岗位专业判断力：质量标准、事实判断、责任边界。
- 相邻岗位协作能力：理解产品、研发、运营、测试、设计等协作对象的输入输出。
- AI 串起小闭环：用 AI 先处理、先预审、先产出，再由人验收。

示例以研发新人展开：读需求、拆任务、写代码、补测试、写说明、看反馈。强调 AI 能补齐协作链，但业务判断和合并责任仍在人。

### AI 新工作语言模块

术语按协同链路重排，不做百科式罗列。每个术语必须包含：

- 解决什么工作痛点。
- 新人怎么用。
- 一句可复制的课堂例子。

保留术语包括 Goal、Context、Prompt、Agent、Eval、Guardrails、Tracing、Loop Engineering、Harness Engineering、Worktree、Subagent。术语分组应服务工作链，而不是形成名词墙。

### 任务卡生成器模块

任务卡生成器从单一表单升级为 4 类模板：

- 周报复盘
- 需求预审
- 竞品调研
- 会议行动项

每类模板提供默认输入、输出结构、验收清单和风险提醒。生成结果是一张完整 AI 协同任务卡，而不只是提示词片段。

### 岗位实战案例模块

删除行政审批预审案例。保留或新增 7 个案例：

- 新人周报
- 需求理解
- 竞品调研
- 代码/文档预审
- 会议纪要
- 运营复盘
- 用户反馈整理

每个案例必须包含：

- 旧做法
- AI 协同做法
- 人负责什么
- 可复制提示词
- 常见失败点
- 验收清单

### 7 天训练计划

7 天行动从“一天一句话”升级为训练表。每天必须包含：

- 训练目标
- 当天任务
- 产出物
- 检查项
- 导师/同伴反馈问题

第 7 天产出自己的第一套 AI 协同工作包，包括任务卡、提示词、验收清单和复盘记录。

## 交互与控件规则

### 保留并做实

- 开始学习：跳到 90 分钟路径。
- 生成任务卡：根据选中模板和输入生成完整任务卡。
- 复制：复制当前任务卡或案例提示词，并显示成功反馈。
- 案例切换：更新案例详情。
- 术语筛选：更新术语列表和详情。
- 全屏展示：保留，按钮状态、图标、焦点样式必须清楚。

### 改造

- 首页术语库圆点和左右按钮不再作为静态装饰。推荐删除首页翻页控件，将主交互放到完整术语模块。
- 下载课程大纲不再下载 README。改成查看 90 分钟路径或复制课堂任务模板等真实课堂动作。
- 泛泛“查看更多”改成明确动作，或删除。

### 删除

- 无效圆点。
- 无效左右箭头。
- 无关 CTA。
- 与课程目标无关的行政审批案例和审批术语示例。

### 控件一致性

- 按钮统一为主按钮、次按钮、图标按钮三类。
- 所有按钮最小高度 44px。
- 所有可交互元素具备 hover、focus、active、disabled 或等价状态。
- 单选/复选使用 segmented chips 视觉，但保留原生 input 语义。
- 复制类按钮必须提供短暂成功反馈，并在 1.5 秒内恢复原标签。
- 键盘 Tab 顺序应与视觉顺序一致。

## 视觉布局规则

- 首屏只承担课程定位、90 分钟安排、最终产出和主要行动，不堆叠工具箱。
- 页面使用 full-width band 分区，每区只用必要卡片承载真实内容。
- 减少一屏内过多边框、阴影和重复卡片。
- T 型模块改成三层能力模型加小流程链路，避免巨大十字造成空白和失衡。
- AI 流程图桌面端使用紧凑横向步骤图，移动端使用纵向步骤。
- 案例详情桌面端两栏：左侧案例列表，右侧详情；移动端列表改为横向 tabs 或紧凑选择区。
- 7 天训练桌面端使用表格/时间线，移动端卡片堆叠。
- 固定宽度改成 minmax、clamp、aspect-ratio 或移动端纵向布局。
- 色彩保留青绿色主色，蓝色用于结构信息，珊瑚色用于风险和边界，避免单一色系。
- 卡片圆角不超过现有 8px 系统。

## 数据与模块边界

继续保持静态站，不引入框架。推荐边界：

- `index.html`：页面结构和静态容器。
- `src/data/workshop.js`：90 分钟路径、课堂产出、7 天训练计划。
- `src/data/cases.js`：岗位案例。
- `src/data/terms.js`：术语和协同链路分组。
- `src/modules/taskCard.js`：任务卡模板切换、生成、复制。
- `src/modules/cases.js`：案例 tabs、案例详情、案例提示词复制。
- `src/modules/terms.js`：术语筛选、详情更新、状态管理。
- `styles/base.css`：设计 token、基础元素、通用交互状态。
- `styles/layout.css`：header、导航、页面分区。
- `styles/sections.css`：课程模块、路径、案例、术语、训练计划。
- `styles/tools.css`：任务卡、表单、chip、复制反馈。
- `styles/responsive.css`：375px、768px、1024px、1440px 响应式规则。

## 测试与验收标准

自动化测试继续使用 `node --test`，新增或更新以下断言：

- 站点运行内容和 README 不包含行政审批相关内容，设计文档可以保留移除背景。
- 90 分钟路径存在，模块时间合计为 90 分钟。
- 7 天训练每天都有目标、任务、产出物、检查项。
- 每个案例都有旧做法、AI 协同做法、人负责什么、提示词、失败点、验收清单。
- 任务卡模板不少于 4 类，并能生成完整任务卡结构。
- 首页不再出现无效轮播控件和下载 README CTA。
- 关键交互 ID 存在：任务模板、生成按钮、输出区、案例区、术语区。
- build 仍能 cache-bust CSS 和 module entry。

实现阶段还需要浏览器验证：

- 375px、768px、1440px 下没有横向溢出。
- 首屏没有过度空白或卡片挤压。
- 所有可见按钮都有真实行为。
- 任务卡可生成、可复制，并有反馈。
- 案例切换和术语筛选可键盘操作。
- 长中文标题和说明不遮挡、不压边、不溢出容器。
- 全屏展示按钮状态可理解。

## 实施风险与处理方式

- 风险：一次性改动较大，容易破坏现有静态 build。
  - 处理：保持原有构建脚本和模块入口，先数据化内容，再替换结构和样式。

- 风险：内容密度增加后页面变长、阅读负担变重。
  - 处理：用 90 分钟路径做主线，案例和术语按需展开，首页不堆所有内容。

- 风险：控件视觉统一后原生表单语义被破坏。
  - 处理：chip 只做视觉包装，保留 input、button、aria 状态和键盘操作。

- 风险：移动端案例和流程图仍可能拥挤。
  - 处理：移动端不横向压缩复杂图，改为纵向步骤或紧凑 tabs。

- 风险：测试只覆盖内容存在，不覆盖真实交互。
  - 处理：单元测试覆盖数据结构和关键容器，浏览器验证覆盖实际交互和响应式表现。
