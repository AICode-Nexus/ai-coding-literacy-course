# 上下文工程管理有限注意力与可信来源

上下文是模型此刻能看到并用于判断的一切：目标、规则、对话、文件、样例、工具结果、计划和状态。窗口能放下更多内容，不代表模型会自动知道什么最重要。

## 本章结论

好的上下文遵循三个原则：

1. **高信号**：优先放入能改变当前判断的事实与约束；
2. **恰当时机**：地图常驻，细节按需检索，阶段结束压缩；
3. **可恢复**：长期任务把意图、事实、进度与分歧写进版本化工件。

> 不要继续问“还能塞什么”，先问“完成下一步判断还缺什么”。

## 知识详解

### 高信号上下文的五层结构

| 层级 | 回答什么 | 典型内容 |
| --- | --- | --- |
| 规则 | 不能违反什么 | 组织政策、数据边界、风格规范、项目原则 |
| 任务 | 这次具体做什么 | 目标、非目标、交付、验收、停止条件 |
| 证据 | 凭什么判断 | 原文、代码、数据、日志、已确认结论 |
| 样例 | 什么样算合格 | 好样例、反例、评分维度、边界情形 |
| 状态 | 现在做到哪里 | 已完成、已验证、阻塞、未决项、下一步 |

每层都应区分“权威、当前、相关”。旧版本、重复材料和未经确认的讨论不能与正式规则处在同一优先级。

### 上下文噪音从哪里来

常见噪音有四类：

- **重复**：同一事实多次出现，占用注意力；
- **冲突**：新旧口径混在一起，模型不知道信谁；
- **无关**：与当前步骤有关联，却不能改变下一步判断；
- **过期**：历史方案、旧状态和已经失效的约束仍被保留。

解决方法不是盲目删除，而是标记来源、版本与优先级，把暂时无关内容移到按需入口。

### 渐进披露：地图常驻，细节按需

长任务可以使用一条上下文供给管线：

`稳定规则 → 任务地图 → 检索入口 → 当前步骤所需证据 → 工具结果 → 阶段状态压缩`

地图说明资料在哪里、谁维护、何时更新；真正需要判断时，再读取对应文件或资源。这样既降低上下文噪音，也保留回到原始来源的能力。

### RAG：为当前问题取回证据

RAG 通常将资料切分、索引，并根据问题取回相关片段。它解决“模型当前看不到组织材料”的问题，但仍可能出现：

- 没有召回关键片段；
- 片段脱离上下文；
- 旧材料排序更高；
- 权限范围错误；
- 模型错误综合已取回内容。

因此 RAG 需要来源、版本、权限、召回检查与最终事实核对，不是“接上知识库就不会幻觉”。

### Compaction：压缩过程，保留可继续工作的状态

普通摘要往往只保留“聊了什么”；任务压缩必须保留“接下来凭什么继续”：

- 当前目标、范围和边界；
- 已确认决策与关键证据；
- 已完成动作和验证结果；
- 失败尝试及其原因；
- 未解决问题、阻塞与下一步。

压缩后的内容应能让新会话、接手人或 Subagent 恢复，而不是重新猜测历史。

### 持久上下文：聊天之外的四类工件

| 工件 | 负责什么 | 更新原则 |
| --- | --- | --- |
| 稳定意图 | 长期目标、原则、责任与边界 | 由人确认，低频变化 |
| 当前工作单元 | 本次 Spec、Plan、决策与验收 | 随任务显式更新 |
| 当前事实 | 代码、测试、数据、版本与运行状态 | 由工具刷新，保留证据 |
| Drift | 新约束、实现偏差、过期假设 | 显式列出，等待对齐 |

机器可以更新可验证事实，但不能静默改写已经确认的意图。发生重大变化时，应明确决定：是事实实现偏了，还是意图正式改变了。

### Project Rules 与 Skill 的上下文边界

AGENTS.md、CLAUDE.md 等项目规则适合保留长期稳定、在项目范围内经常需要的说明。Skill 更适合某类任务触发后按需加载步骤、参考、脚本与资产。

判断方法：

- 每个任务都需要、变化少：项目规则；
- 只有特定任务需要、内容较多：Skill；
- 当前一次任务才需要：任务卡或 Prompt；
- 大量资料只在具体问题出现时需要：RAG、搜索或 MCP Resource。

## 工具与选择方法

<ToolLandscape :category-ids="['deep-research','knowledge-connection','capability-reuse']" />

选择上下文工具时，重点不是“能接多少文件”，而是：

1. 来源是否权威、可追溯并能标记版本；
2. 检索是否能解释命中内容并回到原文；
3. 身份与权限是否跟随真实用户；
4. 更新、失效与删除机制是否清楚；
5. 压缩后能否恢复关键状态。

## 常见误区与边界

### 误区一：长上下文可以替代信息架构

容量只是物理边界。没有优先级、检索与压缩，材料越多越可能产生冲突和噪音。

### 误区二：摘要越短越高信号

删掉来源、失败和未决项的短摘要会让后续工作失去依据。高信号是“能改变下一步判断”，不等于字符少。

### 误区三：RAG 等于组织知识已经正确

RAG 只负责取回候选证据。资料本身的质量、权限、更新和模型综合仍需治理。

### 误区四：机器可以自动维护所有知识

机器适合刷新有明确来源的事实；政策解释、价值判断、责任变化和重大决策需要人确认。

## 岗位迁移

- **研发**：先给仓库地图与项目规则，按需读取相关文件；状态以代码、测试和 Git 为准，意图由 Spec 保留。
- **产品**：区分用户事实、业务决定、讨论意见和待确认假设；研究结论保留来源与日期。
- **运营**：将品牌规则、目标、素材、历史结果与当前发布状态分层；过期素材退出默认上下文。
- **职能**：制度原文优先于二手解释；区分版本、适用范围和办理状态；敏感材料按权限最小化提供。

## 本章练习

对一项任务画“上下文地图”：

| 信息 | 来源与版本 | 常驻 / 按需 / 更新 / 删除 | 为什么影响下一步 |
| --- | --- | --- | --- |
|  |  |  |  |

然后完成两项检查：

1. 删除一项看似相关、但不能改变当前判断的材料；
2. 补上一项缺失的来源、优先级或未决问题。

最后写一份五行状态压缩，看看另一个人是否能据此继续工作。
