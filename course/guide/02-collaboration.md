# 把模糊请求改造成 AI 能做、人能验收的任务

七步法不是提示词公式，而是一张协同检查地图。它既适合在聊天中分步推进，也适合写进 Coding Agent 的任务说明、团队 SOP 或可复用 Skill。

<CollaborationFlow />

## 01 定目标：从动作变成现实变化

“分析”“整理”“优化”“实现”都只是动作。目标应该回答：

- 交付给谁？
- 对方要据此完成什么决定或动作？
- 现实世界发生什么变化才算完成？

**弱目标**：帮我分析一下这份需求。

**可执行目标**：为产品经理和导师输出一份可逐条确认的需求缺口、风险等级和验收标准，帮助评审会决定是否进入开发。

## 02 给材料：提供高信号上下文

材料不在于多，而在于能否支撑下一步判断。优先提供：

1. 可信原文与数据；
2. 当前真实状态；
3. 已确认的历史决策；
4. 一个合格与一个不合格样例；
5. 明确的信息缺口与冲突。

::: tip 让 AI 先报缺口
当材料不完整时，不要让 AI 靠常识自动填空。要求它先列“已知、未知、假设、需要向谁确认”，再决定是否继续。
:::

## 03 签任务：写清动作和顺序

复杂任务不要一次要求完整结果。把工作切成可以检查的中间产物：

`先提取事实 → 再列问题 → 人确认方向 → 再生成方案 → 最后做验收`

好的任务说明包含动作、顺序、使用工具、遇到什么情况暂停，以及怎样报告未完成。

## 04 看产出：先让成果可检查

“给我一份详细分析”缺少可见结构。可以改为：

- 一张证据矩阵；
- 按风险排序的问题清单；
- 可运行代码与变更说明；
- 测试报告和失败样例；
- 一份事实、判断、待确认项分栏的周报。

产出格式不是审美要求，而是后续验证和协作的接口。

## 05 做验收：先定义什么叫好

验收分三层更稳：

<div class="three-layer-check">
  <div><span>LAYER 01</span><strong>工具检查</strong><small>格式、Schema、链接、计算、单元测试、静态规则。便宜且稳定。</small></div>
  <div><span>LAYER 02</span><strong>AI 语义评审</strong><small>完整性、相关性、冲突、风格和开放性质量。需要样例与评分维度。</small></div>
  <div><span>LAYER 03</span><strong>责任人确认</strong><small>业务取舍、授权、安全、对外承诺与不可逆影响。</small></div>
</div>

不要用同一个 AI 在没有独立标准的情况下自问自答。至少提供验收清单、反例或独立证据。

## 06 控边界：权限与副作用要显式

以下动作默认应该暂停并请求确认：

- 对外发送消息、发布内容或提交表单；
- 删除、覆盖或不可逆修改；
- 付费、下单、退款或资源扩容；
- 访问、复制或传播敏感数据；
- 改动生产环境、账号权限和安全设置；
- 对业务、法律、财务、人事等高风险事项做最终判断。

边界不只写“不要做坏事”，还要说明**允许做什么、做到哪一步停、由谁批准、失败怎样恢复**。

## 07 再沉淀：把一次成功变成组织能力

一次任务结束后，至少留下四类资产：

| 资产 | 记录什么 | 下次怎样使用 |
| --- | --- | --- |
| 好样例 | 合格结果及其证据 | 作为 few-shot 或验收参照 |
| 失败样例 | 错误、原因、漏检点 | 加入回归样本集 |
| 检查清单 | 容易遗漏的步骤和边界 | 每次交付前执行 |
| 模板 / Skill | 稳定步骤、资料、脚本 | 渐进加载、重复执行 |

循环必须有预算和停止条件。连续三轮没有实质进展时，应升级给人，而不是让系统无限重试。

## 常见的五种失败

1. **只有角色设定，没有真实目标**：让 AI “扮演资深专家”，却没说成果给谁使用。
2. **材料很多，没有优先级**：过期方案与当前事实同时出现，模型只能猜。
3. **一步交付最终答案**：没有中间产物，方向错误直到最后才暴露。
4. **把主观形容词当验收**：“专业、全面、深入”不能稳定复现。
5. **自动化没有刹车**：工具权限很大，却没有审批、预算和回滚。

## 最小可用任务卡

时间紧时，至少写清以下五行：

```text
目标：给谁交付什么，解决什么问题。
材料：使用哪些可信资料，缺失什么。
交付：先做什么，再按什么结构输出。
验收：至少一条确定性标准和一条人工判断。
边界：哪些动作不能做，遇到什么必须停。
```
