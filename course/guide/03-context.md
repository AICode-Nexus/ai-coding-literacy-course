# 上下文工程管理的是有限注意力，不是字符仓库

上下文（Context）是模型此刻能够看到并用于判断的一切：系统规则、任务说明、对话历史、文件片段、工具返回、样例、当前计划和运行状态。

窗口变大只意味着“能放得下更多”，不保证模型能把关键信号识别得更准。重复、冲突、无关和过期信息仍会稀释注意力。

## Prompt 与 Context 不同

| 概念 | 负责什么 | 常见误区 |
| --- | --- | --- |
| Prompt | 说明这一次要怎样思考与行动 | 以为写得足够长就能补齐缺失事实 |
| Context | 提供完成任务所需的规则、证据和状态 | 把所有资料一次塞入，认为越多越好 |

Prompt 是任务指令的一部分；Context 是完整工作台。一个漂亮的 Prompt 如果没有真实材料，只能让 AI 更有条理地猜。

## Just-in-time：地图常驻，细节按需

长任务中更稳的上下文供给方式是：

1. **常驻最小规则**：安全边界、项目原则、任务目标；
2. **先给信息地图**：有哪些资料、分别在哪里、可信度如何；
3. **按下一步检索**：需要某个判断时，再搜索对应文件或资源；
4. **引用后再行动**：重要结论保留来源，不靠模糊记忆；
5. **阶段结束压缩**：保留结论、证据、状态和未决项，移除冗长过程。

这与 Skills 的“渐进披露”思路一致：先加载名称与适用场景，真正需要时再读取完整说明、参考资料或脚本。

::: warning Compaction 不是普通摘要
压缩必须保留后续行动所需的状态：目标、已确认决策、重要证据、失败尝试、未决问题和下一步。只保留一段自然语言概述，可能让 Agent 忘记关键约束。
:::

## 持久上下文：聊天之外的工作记忆

聊天历史适合帮助思考，但不适合成为项目唯一事实来源。长期任务至少分四层保存：

| 层级 | 内容 | 更新方式 |
| --- | --- | --- |
| Durable intent | 长期目标、原则、边界、技术方向 | 由人确认，低频更新 |
| Work unit | 当前功能或任务的 spec、plan、decision | 随工作单元推进 |
| Current facts | 代码、测试、Git、数据、运行结果 | 工具刷新，反映事实 |
| Drift | 意图与事实不一致、过期假设、新增约束 | 显式列出，等待对齐 |

这个结构来自内部 CDD 实践中的一个稳定洞察：**机器可以刷新事实，但不能静默改写人确认的意图。**

## 不追求“永远一致”，追求偏差可见

在长期项目里，代码、文档、讨论和真实环境一定会变化。强求一份文档始终包含全部最新事实，维护成本很快失控。

更现实的治理方法是：

- 明确哪份工件表达稳定意图；
- 让代码、测试和版本状态表达当前事实；
- 发现不一致时写入 drift，而不是偷偷覆盖；
- 重大变化由责任人 reconcile：确认是实现偏了，还是意图需要正式更新。

> 允许事实变化，但不允许共识悄悄漂移。

## RAG 与 Living Knowledge 的边界

<BoundaryBoard />

RAG（检索增强生成）适合为**这一次问题**找证据片段；活知识页面适合保留团队已经确认的稳定理解、来源和关系。两者组合时，仍需要回到原始证据核验。

<AdvancedTrack title="长程 Agent 的上下文卫生">

长程任务还要管理三种额外噪音：工具返回的大段日志、多个并行任务的中间过程、已经失败但仍留在历史里的旧计划。可以把重日志交给独立 Subagent，只把结论和证据回传主线；每个阶段完成后更新状态工件，并对上下文做结构化压缩。

</AdvancedTrack>

## 本章练习：给你的任务画上下文地图

为你的真实任务列四张清单：

1. **必须常驻**：目标、规则、边界；
2. **按需读取**：原文、文件、网页、历史记录；
3. **每轮更新**：进度、验证结果、未决问题；
4. **必须剔除**：重复、过期、无关或互相冲突的信息。

然后问自己：如果明天换一个新会话，仅凭这些工件，能否在 5 分钟内恢复工作？
