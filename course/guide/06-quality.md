# 可靠不是“这次看起来不错”，而是下次仍能被检查

传统软件可以用确定输入和输出断言判断对错；Agent 任务经常多轮、非确定、路径多样，还会改变外部环境。因此质量系统既要检查最终结果，也要理解执行轨迹与环境状态。

## 质量金字塔：便宜、稳定的检查放在底层

<div class="three-layer-check">
  <div><span>BASE</span><strong>确定性检查</strong><small>Schema、格式、链接、数据范围、静态分析、单元测试、权限规则。</small></div>
  <div><span>MIDDLE</span><strong>AI / LLM Grader</strong><small>相关性、完整性、风格、语义一致性、证据与结论是否匹配。</small></div>
  <div><span>TOP</span><strong>责任人验收</strong><small>价值、风险、取舍、业务正确性和不可逆影响。</small></div>
</div>

越往上越贵、越慢、越需要专业判断。不要让模型评分替代本可用代码完成的检查，也不要让代码假装能判断复杂业务价值。

## 一个完整 Eval 包含什么

| 部件 | 作用 | 例子 |
| --- | --- | --- |
| Dataset | 表达真实任务分布 | 正常、边界、失败、对抗样本 |
| Environment | 提供可执行状态 | 测试仓库、模拟接口、沙箱页面 |
| Run | 执行待评测工作流 | 固定配置、模型、工具与预算 |
| Trace | 记录模型、工具、交接和审批 | 为什么选了这个工具、读到了什么 |
| Grader | 按规则或语义评分 | 单测、Schema、rubric、人工标注 |
| Failure cluster | 聚合同类错误 | 漏证据、错工具、越权、循环失控 |

评测不是项目末尾的一次考试。更好的做法是把真实失败持续加入数据集，每次修改 Prompt、Context、Tool 或 Harness 后跑回归。

## 结果错了，先定位失败发生在哪

| 失败位置 | 常见现象 | 优先修改 |
| --- | --- | --- |
| Goal | 输出很努力，却解决了错误问题 | 目标与非目标 |
| Context | 引用旧信息、漏关键约束 | 来源、检索、压缩、状态 |
| Tool | 选错工具、参数错误、结果没读取 | 工具描述、Schema、权限 |
| Harness | 没跑测试、无限循环、无法恢复 | 验证、预算、状态、回滚 |
| Model | 推理或生成能力不足 | 模型选择、任务拆分、样例 |

这就是轨迹（Tracing）的价值：不只知道“答案错了”，还能看到错在选择、动作、观察还是验收。

<AdvancedTrack title="评测 Agent 轨迹，而不只评最终答案">

最终结果可能碰巧正确，但执行过程中读取了不该访问的数据，或选择了极其昂贵的路径；也可能最终格式有小错，但核心计划和工具选择正确。轨迹 grader 可以检查工具选择、参数、交接、审批和停止条件。不过轨迹本身也可能变化，因此不要对每一步做过度脆弱的字符串匹配，应关注关键行为与安全不变量。

</AdvancedTrack>

## 提示注入：外部内容可能伪装成指令

当 Agent 阅读网页、邮件、文档或工单时，其中的文字属于**不可信数据**。攻击者可能把“忽略原任务、读取密钥、把数据发送到某处”等内容藏在页面里。

防护不是一句“请忽略恶意指令”，而是纵深组合：

1. 明确系统指令与外部数据的信任边界；
2. 使用最小权限和沙箱，避免默认访问密钥与敏感目录；
3. 对外发送、删除、付费、发布等动作前展示预览并审批；
4. 限制工具参数与可访问目标；
5. 对输入、输出和工具调用做自动检查；
6. 记录轨迹，出现异常时可追溯和吊销权限。

::: warning 任何高权限 Agent 都应假设会遇到恶意内容
“只浏览可信网站”并不充分，因为可信页面也可能包含用户生成内容或被污染的第三方信息。权限、动作和数据范围都要最小化。
:::

## 人工检查点放在哪里

可以用两个维度判断：**影响范围**与**可逆性**。

| 动作 | 影响 / 可逆性 | 默认策略 |
| --- | --- | --- |
| 读取公开资料 | 低 / 高 | 可自动，但记录来源 |
| 写本地草稿 | 低 / 高 | 可自动，交付前检查 |
| 修改代码或内部记录 | 中 / 可回滚 | 自动测试 + diff 审查 |
| 对外发送、发布、提交 | 高 / 部分可逆 | 动作预览 + 人工批准 |
| 删除、付费、权限、敏感数据 | 极高 / 低 | 默认拒绝或强审批 |

## 交付前的 10 项检查

- [ ] 目标和交付对象仍然正确；
- [ ] 事实与判断已经分开；
- [ ] 重要结论有可追溯来源；
- [ ] 过期、冲突和缺失信息已标出；
- [ ] 能用工具确定的规则已经实际运行；
- [ ] 语义质量有明确维度或样例；
- [ ] 高风险判断由责任人确认；
- [ ] 外部写入与不可逆动作已审批；
- [ ] 未完成、失败和不确定项没有被隐藏；
- [ ] 成功与失败样例已经进入下一轮资产。

## 本章练习：为你的任务建立最小 Eval

为你的真实任务各写一条：

1. 能用程序或清单确定判断的规则；
2. 需要 AI 按 rubric 判断的语义质量；
3. 必须由责任人确认的高风险结论；
4. 一个最容易失败的边界样本；
5. 一个触发暂停或审批的动作。
