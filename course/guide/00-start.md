# 先给地图，再走路线

这门课的正式目标是：**讲解 AI 思维，帮助校招生掌握“怎么用 AI 提效、用什么思路用 AI”，形成“人机协作”的工作思维。**

这里的“提效”不是终点。AI 更深一层的价值，是**释放人的想象力、扩大人的执行力**：让人有能力探索更多可能，把构想更快变成候选、原型和可验证成果，并在真实反馈中加速创新。

它不以背诵工具功能、模型名称或 Agent 术语为终点。学完的标志是：你能面对一项工作，说明目标与责任，组织可信材料，让 AI 参与合适的步骤，用证据验收结果，并把有效做法沉淀给下一次。

## 为什么从“会用”继续往前走

内部[学员统计材料](/sources#internal-learner-stats)显示，参与群体并不是 AI 零基础：通用模型和 AI Coding 已有较高使用基础，而 Agent、工作流与多模态实践相对较少。具体数据和教学含义放在[第 02 章](/guide/02-collaboration-shift)展开；课程地图只用它确定“不再从零教工具按钮”的起点。

这些数据不能推断每个人的岗位与熟练度。材料没有可读取的完整岗位明细，因此本课程不假定所有学员都是研发岗，也不把“用过某类工具”等同于已经形成稳定协同方法。

真正需要补齐的是从**单点动作**到**完整工作链**的迁移：

| 已经会做 | 还要继续建立 | 结果变化 |
| --- | --- | --- |
| 提问、总结、润色 | 目标与交付对象 | 从一段回答变成有用途的成果 |
| 搜索、生成初稿 | 来源与上下文 | 从常识猜测变成证据支持 |
| 使用 AI Coding | 测试、权限、状态与回滚 | 从生成代码变成可验证改动 |
| 使用工作流或 Agent | 停止条件、评测与人工接管 | 从能运行变成可控运行 |
| 有想法但验证成本高 | 候选发散、快速原型与反馈 | 从“想得到”变成“做得出、验得快” |

## 九节点、80 分钟主讲路线

<CourseRouteMap />

::: info 第一章和第二章为什么都保留
两章会共用 Goal、Context、Eval 等基础词汇，这是有意保留的“同一套语言”，不是要再讲一遍同样的知识。**第一章讨论人才能力边界**：怎样守住专业纵深，并向前后各跨一步形成相邻小闭环；**第二章讨论 AI 使用成熟度**：为什么会用工具后仍然拉开差距。共用词汇只做索引，每章的证据、主问题、案例和练习各自展开。
:::

### 九章各自只回答一个主问题

| 节点 | 本章主问题 | 不在本章重复展开 |
| --- | --- | --- |
| 01 T 型人才 | 个人的专业深度与横向协作边界在哪？ | 不讲完整任务流程 |
| 02 人机协同 | 为什么用了同样的 AI，结果差距仍然很大？ | 不提前教完七步法 |
| 03 基础全景 | 任务到底需要哪类模型与工具能力？ | 不把产品排名当方法 |
| 04 协同七步法 | 怎样把模糊要求改写成可执行任务？ | 不深入讲评测工程 |
| 05 Context | 模型此刻应该看见什么，不该被什么干扰？ | 不展开 Agent 编排 |
| 06 Agent | 什么时候值得从回答进入受控行动？ | 不以 Agent 数量代替系统质量 |
| 07 质量与安全 | 凭什么相信结果，谁为后果负责？ | 不只做任务卡层面的验收 |
| 08 流程重构 | 一次成功怎样进入下一次工作？ | 不重新教一遍七步法 |
| 09 岗位迁移 | 未来七天怎样在自己岗位跑通第一个小闭环？ | 不再建第二套方法 |

主讲结束后保留 **15 分钟交流**。建议优先讨论学员带来的真实任务：目标是否可观察、材料从哪里来、怎样验收、哪里必须停下来，而不是把时间变成产品功能问答。

## 全课主方法只有一张图

<CollaborationFlow />

七步是一张检查地图，不是强制每次写七段长提示词。轻任务可以只写目标、交付和验收；复杂或高风险任务才需要完整展开材料、动作、权限、停止条件和沉淀方式。

遇到结果不稳定时，先定位问题所在：

1. **目标层**：是不是解决了错误问题？
2. **信息层**：关键事实是否缺失、冲突或过期？
3. **行动层**：工具、权限与环境是否合适？
4. **质量层**：通过标准、证据与责任人是否明确？
5. **复用层**：下一次是否还要重新解释和重新检查？

## 六个主问题先认路

课程地图不先堆概念。你只要记住六个问题：要造成什么变化？凭什么判断？交付什么？怎样证明通过？哪里必须停？下次怎样更好？第 03 章再给出模型、工具与行动系统的完整图。

## 工具先按能力分类

课程不发布工具排行榜，不固化价格、上下文窗口数字或“最强模型”结论。先按任务需要选择能力类别，再查看当前代表入口。

<ToolLandscape mode="compact" />

正式使用某个产品前，还要重新确认组织政策、数据边界、可用版本、价格和能力状态。工具条目带核验日期；超过复核周期后会显示“待重新核验”。

## 四份课程成果

1. **一张 AI 协同任务卡**：目标、对象、材料、动作、交付、验收和边界完整可见。
2. **一份三层验收清单**：确定性检查、AI 语义检查与责任人确认各司其职。
3. **一个岗位小流程改造方案**：明确哪些步骤删除、合并、交给 AI、保留给人。
4. **一份 7 天行动计划**：在真实工作中跑完一个低风险、可验收的小闭环。

## 内容怎样保持不过时

课程把知识分成三类维护：

- **稳定方法与协议**：概念定义、协同结构、质量与安全原则，优先使用官方一手资料；
- **动态产品入口**：模型、工具与平台只作为能力示例，记录核验日期与复核周期；
- **内部材料**：只用于课程目标、学员起点和组织语境，不扩写无法核验的项目故事。

如果内容缺少来源，就只能明确标记为“场景演示”或“应用示例”，不能包装成已经发生的事实。完整记录见[来源与核验](/sources)。
