# 先看任务约束，再选择能力组合

工具更新很快，产品能力也会重叠。比背诵一份名单更重要的是：理解模型的基本能力与限制，识别任务需要什么，再用稳定维度比较当前选择。

## 本章结论

::: tip 本章核心观点
面对一项任务，先回答六个问题：

1. 任务需要简单生成还是复杂推理？
2. 上下文规模与来源结构怎样？
3. 是否需要理解或生成图像、音频、视频？
4. 只要内容，还是需要搜索、读写、执行与反馈？
5. 数据安全、组织政策与权限边界是什么？
6. 总成本是多少，包括等待、返工、上下文准备和风险？

然后选择能力组合，最后再看代表产品。**没有长期不变的最强模型，也没有适合所有任务的单一工具。**
:::

模型基础层可以连成一条能力链：

`LLM 通过 Token 处理 Context → Reasoning 为复杂问题分配更多计算 → Multimodal 扩展可读取和可生成的信息形态`

Hallucination 始终提醒我们：语言流畅、推理更长、窗口更大或支持更多模态，都不代表事实自动可靠。

## 知识详解

### LLM：强大的生成与模式能力，不是事实数据库

大语言模型依据训练与当前上下文生成后续内容。它擅长理解任务说明、转换表达、归纳材料、比较选项、生成结构和代码，也可能在缺少证据时给出非常流畅的错误内容。

因此模型输出至少要区分：

- **来自材料的事实**：可以回到来源；
- **基于事实的判断**：要说明依据与不确定性；
- **模型生成的建议**：需要责任人评估是否适用；
- **仍缺失的信息**：不应被常识自动填平。

### Token 与 Context Window：容量不等于注意力

Token 是模型处理信息的基本片段，不等同于固定汉字数。Context Window 表示一次运行可接收的指令、对话、文件片段和工具结果总量。

更大窗口有助于处理长材料和长任务，但仍有四种成本：

1. 无关内容稀释关键信号；
2. 冲突版本让模型难以判断优先级；
3. 长轨迹增加延迟与调用成本；
4. 重要信息可能藏在难以关注的位置。

所以长上下文要与检索、排序、来源标注和压缩一起设计。

### Reasoning：给复杂问题更多计算，不会修复错误前提

推理能力适合多约束规划、复杂分析、数学、代码与需要比较路径的任务。它通常用更多时间与计算换取更完整的处理。

但如果材料错误、任务目标含糊、缺少必要工具或验收标准错误，更多推理只会更认真地解决错误问题。模型选择之前，先保证问题值得被解决。

### Multimodal：共同处理多种材料，也要处理授权与事实

多模态能力让模型理解截图、图表、语音、视频或生成视觉内容。使用时除效果外还要检查：

- 输入是否包含隐私、肖像或未授权材料；
- 模型是否真的读取了关键视觉细节；
- 生成内容是否可能误导事实；
- 输出尺寸、格式、版权与后期可编辑性是否满足交付。

### Tool 与 Agent：从“知道”走向“做到”

Tool 是搜索、文件、终端、浏览器或业务接口等结构化动作。Agent 则围绕目标选择工具、读取反馈并决定继续或停止。

如果任务只需要解释与草稿，通用助手可能已经足够；如果需要最新来源、修改真实对象、运行测试或跨系统执行，就要评估工具与 Agent。能力越接近真实环境，权限、日志、审批和回滚越重要。

### 六维模型选择法

| 维度 | 要问的问题 | 常见错误 |
| --- | --- | --- |
| 任务难度 | 是直接生成、归纳，还是多约束推理？ | 所有步骤都用最贵模型 |
| 上下文 | 材料多少、来源怎样、是否频繁更新？ | 只看窗口数字，不做信息设计 |
| 多模态 | 要理解还是生成哪些模态？ | 支持图片就假设能做专业视觉判断 |
| 工具能力 | 是否需要搜索、文件、代码或业务动作？ | 把会回答当成能执行 |
| 数据安全 | 能否上传、存储、连接与留痕？ | 个人便利覆盖组织政策 |
| 总成本 | 费用、等待、准备、返工、复核与风险？ | 只比较单次价格 |

总成本不是“越便宜越好”。如果一个工具首轮输出快，却需要大量核对、复制、格式修复和权限补救，真实交付成本可能更高。

组合工具时遵循最小充分原则：

- 一项能力能完成，就不急着叠加平台；
- 需要最新证据，增加研究与知识连接；
- 需要改变真实对象，增加 Tool / Agent 与 Harness；
- 任务稳定重复，才进入工作流与能力复用；
- 任何组合都要保留验收、权限和人工责任。

## 常见误区与边界

### 误区一：排行榜第一就适合我的任务

公开评测只能覆盖特定数据集与配置。你的材料、工具、语言、风险和验收可能完全不同。用代表任务做小规模对比更可靠。

### 误区二：参数更大、推理更久，结果一定更好

能力与成本不是线性关系。简单步骤可能因为过度推理变慢或偏离格式；关键是任务匹配与可验证结果。

### 误区三：产品支持长上下文，就可以一次上传全部资料

容量不能替代来源治理、冲突处理和渐进披露。敏感资料还要先判断是否允许进入该服务。

### 误区四：带引用的研究结果可以直接使用

引用存在不等于引用支持结论。应打开关键来源，核对发布时间、原文语境、数据范围与相互冲突。

### 误区五：工具列表一旦写进教材就长期有效

型号、入口、价格和功能都可能变化。本课程只把代表产品当作动态示例，并通过核验日期与复核周期提醒更新。

## 岗位迁移

- **研发**：重点判断仓库理解、终端与测试、权限沙箱、差异审查与回滚；不要只比较代码补全效果。
- **产品**：重点判断研究来源、长材料处理、结构化输出、多方案比较和敏感业务信息边界。
- **运营**：重点判断图文音视频能力、编辑可控性、品牌事实检查、素材授权与交付格式。
- **职能**：重点判断文档处理、制度来源、数据隐私、组织批准与是否能保留可追溯记录。

## 本章练习

选择一项任务，用六维表填写：

```text
任务难度：________________
上下文规模与来源：________________
多模态需求：________________
需要的工具动作：________________
数据与权限边界：________________
总成本中最可能被忽略的一项：________________
```

再从七类能力中选择**最少的一组**，并写出不选择其他类别的理由。最后定义一个小样本：怎样比较两种候选能力组合，而不是只看宣传页。
