# 差距从工具熟练度转向任务组织能力

大家都会提问、总结、润色或生成初稿后，为什么结果仍然差很多？因为“得到一段内容”和“推动一项工作完成”需要不同的组织能力。

## 本章结论

::: tip 本章核心观点
工具使用关注一次交互：我怎样问、它怎样答。人机协同关注完整任务：结果给谁、材料从哪里来、怎样执行、怎样证明完成、哪里不能越界、下次怎样复用。

> 模型能力决定可能的上限；任务组织方式决定日常可重复的下限。

从工具到协同不是放弃聊天框，而是把聊天放回工作链。简单任务仍然可以一句话完成；一旦涉及多人、证据、工具或风险，就要补齐协同结构。

人机协同也不只服务于“把原有任务做快”。当想象、原型、验证和反馈被组织进同一条链，AI 能帮助人扩大探索空间、增强执行能力，让创新更快从一个念头进入真实世界。
:::

## 概念与关系

<KnowledgeAtlas mode="compact" :concept-ids="['prompt','task-contract','loop','tool','agent']" />

第 01 章已经出现的 Goal、Context、Output、Eval 和 Guardrails，是全课通用的协同骨架，这里不再把它们展开成第二套卡片。本章只新增五个“从问答迈向系统”的识别信号：

- **Prompt** 是这一次运行的说明；
- **Task Contract** 让人、AI 和验收者共享完成条件；
- **Tool** 把判断转成一次真实动作；
- **Agent** 围绕目标组织多步动作与反馈；
- **Loop** 根据结果继续、修正、停止或交还给人。

因此，一个表达漂亮的 Prompt 如果缺少来源、工具、验收或责任人，仍然无法稳定交付。

## 知识详解

### 先看学员起点：“用过”与“用好”之间还有一段距离

| 内部学员数据 | 能够说明 | 不能说明 |
| --- | --- | --- |
| 超过 90% 使用过通用大模型 | 聊天、生成与问答不再陌生 | 已经能定义任务、核对来源或承担结果 |
| 78.21% 使用过 AI Coding | 代码生成与辅助编程有较广接触 | 已经会读仓库、控制改动、运行测试和审查差异 |
| 21.23% 使用过 Agent 或工作流 | 一部分学员接触过多步执行 | 已经解决了权限、日志、停止、验收和失败恢复 |
| 25.70% 接触过 AI 图像或视频 | 多模态不是完全陌生的方向 | 已经处理了事实、版权、肖像、格式与交付质量 |

这组数据的教学用途只有一个：课程不必再证明“AI 能生成内容”，而要让学员看见，接触过某类产品，与能否把它组织进真实工作，是两件事。

### 四级使用成熟度

| 层级 | 典型状态 | 主要价值 | 最常缺什么 |
| --- | --- | --- | --- |
| 1 问答与生成 | 得到解释、摘要、草稿 | 减少空白启动 | 事实与用途 |
| 2 辅助完成任务 | AI 参与某一段真实工作 | 局部提速、快速试探 | 完成标准与上下游 |
| 3 进入可验收流程 | 有材料、步骤、检查和接管 | 稳定交付、验证新方案 | 复用与可观察性 |
| 4 形成可复用系统 | 模板、规则、工具和评测持续更新 | 组织能力 | 治理与失效机制 |

成熟度描述的是**某类任务**，不是给人贴高低标签。同一个人写短消息可能处于第 1 级，处理高风险发布可能需要第 3 级，维护成熟自动化流程时才进入第 4 级。

::: tip 本章只做“定位”
这四级用来判断一项任务当前处在哪里，不在本章把每级的实现细节都教完。怎样写任务契约由第 04 章负责，怎样构造 Agent 由第 06 章负责，怎样建评测体系由第 07 章负责。
:::

### 差距为什么会在“都会用”之后继续扩大

第一层差距是**问题组织**。有人把一句要求直接交给模型，有人会先确认对象、变化、限制与完成证据。两人即使使用同一模型，实际上也在解不同质量的问题。

第二层差距是**证据链**。没有来源的人只能评价“写得像不像”；把原文、数据、代码、工具结果和测试日志带进来的人，才能评价“这件事是否成立”。

第三层差距是**反馈与资产**。一次性用法把失败留在聊天记录里；协同用法把失败转成新样例、检查项、规则或流程调整。所以真正的差距不只发生在第一次输出，还会在每一轮是否学会中累积。

### 一次性回答为什么容易失效

一次性请求把许多隐含判断交给模型猜：

- 谁是最终使用者；
- 哪些材料权威且仍然有效；
- 哪些内容是事实、假设或待确认项；
- 交付格式怎样支持下一步工作；
- 怎样判断“完成”；
- 发生什么必须停止。

模型可能用常识补齐这些空白，并生成一段结构完整的回答。问题是：结构完整会掩盖前提缺失，让使用者更晚发现方向错误。

### 协同系统不是每一步都自动化

一条健康的协同链通常包含：

`责任人定义结果 → AI 暴露缺口 → 人补充关键材料 → AI 形成中间产物 → 工具执行确定动作 → 机器与 AI 分层检查 → 人确认高风险判断 → 经验进入下一轮`

人在链路中出现多次，不是因为 AI “不够先进”，而是不同判断的证据、责任与可逆性不同。自动化的目标是减少无意义搬运和重复检查，不是把责任人从流程图上删除。

### 从内容结果到工作结果

判断 AI 是否真正进入工作，可连续追问：

1. 这份输出交给谁？
2. 对方会用它做什么决定或动作？
3. 哪些结论能回到来源、测试或运行结果？
4. 不通过时能定位到哪一步？
5. 下一次是否少解释、少返工或少检查？

如果回答只停在“写得更快、语言更好”，那仍属于内容提效；如果成果能被使用、验证和复用，才开始形成工作系统。

### 提效之后，还要看创新是否被加速

创新加速不是让 AI 一次生成更多点子，而是缩短三段距离：

1. **从问题到可能性**：AI 帮助发散假设、连接不同领域、暴露被惯例忽略的选项，释放人的想象空间；
2. **从可能性到原型**：把文字构想快速变成草图、流程、代码、样稿或其他可检查的中间产物，扩大人的执行力；
3. **从原型到反馈**：用来源、测试、用户或责任人确认尽早淘汰错误方向，让有价值的想法更快迭代。

因此，**提效关注同一件事能否更快完成，创新加速关注原本难以尝试的想法能否更早被做出、验证和改进。** 两者都需要证据、边界和责任人，否则“生成得更多”只会制造更多待筛选内容。

### 人与 AI 的动态分工

| 工作类型 | 更适合 AI 扩大 | 更需要人负责 |
| --- | --- | --- |
| 信息处理 | 搜索、提取、比较、结构化 | 来源优先级、遗漏影响 |
| 方案形成 | 发散、列选项、模拟反对意见 | 目标、取舍、资源承诺 |
| 执行动作 | 生成、调用工具、批量处理 | 授权、异常、不可逆影响 |
| 质量检查 | 规则检查、语义预审、失败聚类 | 业务正确性、风险与最终签字 |

分工不是固定角色表。任务风险、证据形式和工具能力变化时，检查点也应随之变化。

## 工具与选择方法

不同成熟度对应不同能力组合。不要因为工作流平台或 Coding Agent 看起来更先进，就跳过任务契约和验收。

<ToolLandscape mode="compact" :category-ids="['general-assistant','coding-agent','agent-workflow','capability-reuse']" />

选择顺序建议是：

1. 通用助手能否完成低风险、可检查的部分？
2. 是否真的需要操作仓库、文件或业务系统？
3. 多步骤动作是否稳定重复，值得进入工作流？
4. 是否已经定义日志、失败、审批与恢复？
5. 哪些说明和检查可以沉淀为复用资产？

## 常见误区与边界

### 误区一：协同就是写更长的 Prompt

长 Prompt 只能表达更多指令，不能自动提供缺失事实、真实工具结果和责任人确认。先判断缺的是表达、信息、行动环境还是反馈机制。

### 误区二：AI 参与步骤越多越好

不必要的 AI 环节会增加延迟、成本、信息损失和验证负担。只在能减少断点或扩大有效检查面的地方加入。

### 误区三：用了 Agent 就自然进入系统级

Agent 可能只是把多个不清楚的步骤自动串起来。如果目标、权限、验收与停止条件不清，自动化只会更快放大错误。

### 误区四：人机协同意味着人逐字复核全部内容

应把确定性错误交给规则与工具，开放语义交给 AI 初审，人的注意力集中在价值、风险和责任上。

## 岗位迁移

- **研发**：从“生成代码”迁移到“理解仓库—修改最小范围—运行测试—审查差异—说明风险”。
- **产品**：从“生成文档”迁移到“明确决策问题—组织证据—暴露未知—形成可确认项”。
- **运营**：从“生成文案”迁移到“目标与受众—素材来源—多版本产出—事实与品牌检查—发布审批”。
- **职能**：从“润色材料”迁移到“规则核对—材料缺口—结构化交付—隐私与审批—服务结果”。

## 本章练习

选一次最近的 AI 使用，画出它的工作链：

```text
任务从哪里来 → 我提供了什么 → AI 做了什么 → 谁检查 → 谁使用 → 留下什么
```

然后只补一个最缺的环节：

- 目标不清：补交付对象与完成状态；
- 材料不足：补来源、版本与缺口；
- 结果难验：补结构与通过标准；
- 风险不明：补权限、停止与人工接管；
- 每次重来：补模板、样例或检查清单。
