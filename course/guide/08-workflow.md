# 流程重构既释放效率，也加速创新

一次回答更快，只改变了当前任务；模板、规则、检查和状态进入下一轮，才开始改变长期工作方式。流程重构关注的不只是“哪一步加 AI”，而是哪些步骤应该消失、合并、重排或重新分工，以及怎样让新想法更快获得原型和反馈。

## 本章结论

::: tip 本章核心观点
从一次提效走向流程能力，要完成两个循环：

- **交付循环**：目标—执行—验收—完成；
- **学习循环**：复盘—提取资产—版本化—评测—回到下一次。

两个循环稳定后，重复劳动被压缩，人可以把更多注意力用于提出新问题和做创造性取舍；AI 则把构想更快转成候选、原型和验证动作。真正的指标不是 AI 参与了多少步骤，而是下一次是否更少解释、更少等待、更少返工、更早发现风险，以及一个新想法从提出到获得首次有效反馈的距离是否缩短。
:::

## 通用工作场景

<ScenarioFrame scenario-id="workflow-reuse-or-restart" />

场景不要求建设大型平台。个人可以从一份模板和清单开始，团队可以逐步增加规则、Skill、知识连接和自动化。复用复杂度要随真实重复频率增长。

## 概念与关系

<KnowledgeAtlas mode="compact" :concept-ids="['loop','skill','project-rules','harness','eval','ai-maintained-wiki','loop-engineering']" />

Output 是一次交付；Loop 把反馈带回下一轮；Project Rules 与 Skill 保存稳定做法；Eval 判断修改是否真的更好；AI-maintained Wiki 与 Loop Engineering 探索让事实与流程持续更新的方式。

第 04 章和第 08 章都会出现“沉淀”，但产物不同：

| 对比 | 第 04 章：完成当前任务 | 第 08 章：改造下一次工作 |
| --- | --- | --- |
| 关键问题 | 这次怎样做清楚、做完并验收？ | 哪些做法真的稳定，值得进入下一次？ |
| 起点 | 一句要求与当前材料 | 多次真实执行、成功与失败证据 |
| 核心工件 | 任务卡、中间产物、验收记录 | 模板、规则、样例、检查、Skill、状态与评测集 |
| 通过标准 | 当前结果能被使用且证据齐全 | 下一次少解释、少返工，且没有固化新风险 |
| 失效处理 | 返回对应步骤补齐缺口 | 更新、降级或退役资产，并增加回归样例 |

## 知识详解

### 复用阶梯：不要第一天就自动化

| 层级 | 适合何时开始 | 主要工件 | 风险 |
| --- | --- | --- | --- |
| 好样例 | 第一次成功后 | 合格输出与来源 | 只学表面格式 |
| 检查清单 | 发现重复错误后 | 通过项、失败项、停止项 | 清单失效不更新 |
| 模板 | 任务结构稳定重复 | 字段、说明、空白样板 | 让人跳过必要判断 |
| 项目规则 | 多任务都需要同一约束 | 原则、目录、命令、边界 | 过长占用上下文 |
| Skill | 某类任务流程可复用 | 指令、参考、脚本、资产 | 偶然成功过早固化 |
| 自动化与持续评测 | 高频、稳定、收益明确 | 编排、状态、日志、评测集 | 失败分支与治理不足 |

每升一级，都要补充适用边界、版本、负责人和验收。复用不是把旧答案复制得更快，而是让稳定方法可被正确触发、执行和检查。

### 从聊天记录提取真正资产

聊天全文很少是高质量资产。复盘时应提取：

- 有效目标表达与必要字段；
- 真正改变判断的材料与来源；
- 成功步骤和不应重复的失败；
- 确定性检查与语义评分维度；
- 需要人确认的判断；
- 停止、升级和恢复方式。

资产必须可独立理解。离开原对话后，如果不知道何时使用、输入是什么、怎样通过，就还不能复用。

### 流程重构的五个观察点

先画现状，不急着加工具。标出：

1. **重复**：同样的信息被多次复制、整理或解释；
2. **等待**：任务卡在交接、确认或系统切换；
3. **返工**：错误发现太晚，或完成标准一开始不清；
4. **高风险**：权限、发布、敏感数据与不可逆动作没有检查点；
5. **探索受阻**：新想法只能停在讨论里，做原型或获得反馈的成本太高。

然后重新分配：

- 删除没有价值的搬运；
- 合并可以共享同一上下文的步骤；
- 把确定性动作交给代码或工具；
- 用 AI 同时扩大候选空间、原型速度与开放性检查能力；
- 把授权、取舍和高风险确认留给人；
- 把状态、证据和失败写回资产。

流程因此释放两类空间：一类是减少重复劳动得到的时间，另一类是把原本做不起、试不快的想法变成低成本实验的创新空间。前者让人有余力想象，后者让人有能力把想象推进到现实反馈。

### 资产循环：工作产生知识，知识改善工作

一条可持续循环可以写成：

`任务 → 执行 → 验收 → 复盘 → 模板 / 规则 / 样例 / 检查 / 状态 → 版本化与评测 → 下一任务`

成功样例告诉系统什么有效，失败样例告诉系统哪些边界不能再犯。只有两者都进入评测，流程才不会只积累“最佳实践”口号。

### AI-maintained Wiki：自动更新事实，人工守住意图

AI 可以根据代码、测试、工单或权威来源刷新事实页面、发现过期内容和生成变更建议。更稳的边界是：

- 可验证事实可自动刷新；
- 来源、版本和更新时间必须保留；
- 稳定意图、政策解释和责任变化由人确认；
- 不确定或冲突内容进入 Drift，而不是静默覆盖；
- 页面使用情况与错误反馈进入评测。

### Loop Engineering：把工作循环本身当成产品

Loop Engineering 关注的不只是单个 Prompt，而是整个“输入—行动—反馈—状态—改进”循环。设计对象包括：

- 哪个状态决定下一步；
- 什么反馈可以自动读取；
- 哪种失败值得重试，哪种必须升级；
- 经验怎样进入规则、Skill 或知识；
- 如何用评测确认流程没有退化。

这是前沿实践，不要求所有学员马上建设。A 类学员先从复盘模板和检查清单开始，就已经进入正确方向。

<AdvancedTrack title="B 类进阶：为复用资产建立失效机制">

每个模板、规则或 Skill 都应记录负责人、适用范围、版本、最后核验日期与代表任务集。当来源变化、连续失败或长期无人使用时，触发复核或退役。没有失效机制的知识库会把过期经验稳定地放大。

</AdvancedTrack>

## 工具与选择方法

<ToolLandscape mode="compact" :category-ids="['agent-workflow','knowledge-connection','capability-reuse']" />

决定是否进入平台化前，先问：

1. 任务是否高频且结构稳定？
2. 输入、输出和通过标准是否已经清楚？
3. 失败分支、人工接管与恢复是否设计？
4. 数据与权限能否按真实身份治理？
5. 资产是否能导出、版本化与迁移？

如果这些问题还没有答案，先用模板、清单和手动闭环验证，不急着画复杂工作流。

## 常见误区与边界

### 误区一：自动化就是流程重构

自动化旧流程可能把无价值步骤一起固化。先删除、合并和重排，再决定哪里自动。

### 误区二：保存 Prompt 就完成沉淀

可复用资产还需要适用场景、输入要求、边界、检查、版本和负责人。

### 误区三：知识越多，组织能力越强

没有更新、检索、使用和退役机制的知识会变成噪音。质量与可用性比数量重要。

### 误区四：一次成功就适合封装 Skill

先在多个代表任务上验证，记录失败与人工判断，再提取稳定部分。

### 误区五：全自动是最终目标

高价值协商、专业取舍和高风险动作可以明确保留给人。目标是责任清楚与流程有效。

## 岗位迁移

- **研发**：把稳定命令、项目规则、测试和代码审查清单沉淀；高频改动再封装 Skill 或 CI，保留发布与安全审批。
- **产品**：沉淀需求预审模板、决策记录、研究来源与问题清单；不要把历史讨论自动当成当前决定。
- **运营**：沉淀素材规范、事实检查、品牌清单与复盘结构；发布效果进入下一轮，但相关不等于因果。
- **职能**：沉淀标准表单、材料清单、制度来源、审批点与风险清单；规则版本与适用范围必须可追溯。

## 本章练习

选择一个已经重复三次的任务，完成“现状—目标”对照：

```text
重复搬运：________________
等待与交接：________________
最晚才发现的返工：________________
高风险动作：________________

准备删除的步骤：________________
交给工具的确定动作：________________
交给 AI 的开放任务：________________
保留给人的判断：________________
准备沉淀的第一个资产：________________
```

只选择一个轻量资产开始：模板、清单、好坏样例或脚本。为它写上适用边界和失效条件。
