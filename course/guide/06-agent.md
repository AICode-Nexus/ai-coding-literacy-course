# 默认最简单结构，按独立性与边界增加 Agent

普通问答大多是输入后生成内容。Agent 则围绕目标选择动作、调用工具、读取结果、检查状态并决定继续、停止或交还给人。

## 本章结论

Agent 的价值不在“像人一样自主”，而在于把多步骤行动与真实反馈组织进受控循环。

默认使用能完成任务的最简单结构：

- 一个主线可以完成：Single Agent；
- 子任务独立、上下文需要隔离：Subagent；
- 多个角色必须直接协商：谨慎评估 Agent Team。

每增加一个 Agent，都要同时增加输入输出契约、通信、验证、冲突处理和成本治理。

## 通用工作场景

<ScenarioFrame scenario-id="agent-orchestration-choice" />

复杂不等于适合并行。判断的关键是子任务是否独立、能否单独验收、上下文或权限是否需要隔离，以及协调成本是否小于等待收益。

## 知识详解

### Agent Loop 的六个部件

`Goal → Plan → Tool → Observation → Check → Continue / Stop / Handoff`

1. **Goal**：完成状态明确，否则循环不知道何时结束；
2. **Plan**：选择当前最小下一步，而不是一次生成全部未来；
3. **Tool**：动作空间受限，参数结构清楚；
4. **Observation**：读取文件、测试、页面或接口的真实结果；
5. **Check**：按标准判断通过、修复、暂停或升级；
6. **Stop / Handoff**：达到目标、触发风险、无进展或预算限制时退出。

Agent 不是某个模型名称。相同模型放在不同工具、权限、测试与状态环境中，会表现出完全不同的工作能力。

### MCP：连接协议，不是智能体

MCP 定义主机、客户端与服务器怎样交换能力。常见原语包括：

- **Tools**：模型可调用的动作；
- **Resources**：应用可读取的上下文；
- **Prompts**：服务器提供的可复用交互模板。

协议降低重复接入成本，但不会自动证明服务器可信、数据可用或动作安全。身份、权限、来源、批准和审计仍由系统设计负责。

### Skill：把做法变成按需加载的资产

一个成熟 Skill 可以包含：

- 何时使用与何时不使用；
- 经过验证的步骤与检查点；
- 领域参考、模板、好样例与反例；
- 确定性脚本与交付资产；
- 失败处理、版本和评测方式。

一次偶然成功的 Prompt 不应立即封装。先用真实任务重复验证，再决定哪些步骤稳定、哪些判断仍要留给人。

### Single、Subagent 与 Agent Team

| 结构 | 适合 | 主要收益 | 主要成本 |
| --- | --- | --- | --- |
| Single Agent | 一个主线、上下文共享、步骤强依赖 | 简单、状态一致、易调试 | 主线可能积累噪音 |
| Subagent | 独立研究、重日志、可单独验收 | 上下文隔离、可并行 | 委派与汇总损失信息 |
| Agent Team | 成员需直接讨论、挑战与协商 | 动态协作与多视角 | 通信、冲突、等待与 Token 高 |

Agent Team 在部分产品中仍是实验性能力。不要为了“多智能体”标签使用它；只有角色之间必须直接协商，且收益足以覆盖成本时才值得尝试。

### 先画依赖，再决定并行

通常适合并行：独立资料核验、互不影响的测试、不同来源的扫描、可单独审查的模块。

通常保持顺序：先设计后实现、下一步依赖前一步结论、同一文件密集修改、需要频繁共识更新。

每个并行子任务都应写清：输入、非目标、交付、验收、返回格式、允许修改范围和谁负责合并。

### Harness Engineering：把能力关进可控环境

Prompt 说明这次怎样思考；Harness 决定 Agent 是否有条件安全做成。常见部件包括：

- 可发现且描述清楚的工具；
- 最小权限与动作审批；
- 沙箱、工作目录或隔离 Worktree；
- 测试、Schema 与静态规则；
- 轨迹、预算、超时和停止条件；
- 状态保存、差异审查与回滚。

非代码任务也需要 Harness：草稿区代替直接发布，副本代替覆盖原文件，预览代替直接发送，审批代替隐式授权。

### Worktree：隔离修改，不等于自动合并

Git Worktree 可以让不同任务在独立工作目录和分支中进行，降低文件争抢和状态污染。它适合边界清楚的并行改动，但仍需：

- 明确分支与文件所有权；
- 在合并前同步主线变化；
- 运行完整测试与审查差异；
- 处理逻辑冲突，而不仅是文本冲突。

<AdvancedTrack title="B 类进阶：Agent 编排的三个控制面">

**任务面**定义依赖、输入输出与完成状态；**上下文面**决定每个 Agent 看到什么、怎样压缩与汇总；**治理面**控制工具、权限、预算、审批、日志与回滚。只做任务分解而忽略后两个控制面，往往得到“看起来并行、实际上难以验证”的系统。

</AdvancedTrack>

## 工具与选择方法

<ToolLandscape :category-ids="['coding-agent','agent-workflow','knowledge-connection','capability-reuse']" />

选 Agent 或平台时，优先检查：

1. 能否清楚显示它读了什么、做了什么、为什么继续；
2. 工具与权限能否最小化并按动作审批；
3. 测试、日志、预算、停止与恢复是否存在；
4. 状态能否跨会话或跨人员恢复；
5. 是否能从单 Agent 起步，而不是被迫使用复杂编排。

## 常见误区与边界

### 误区一：能调用工具就是 Agent

单次 Tool Use 只完成一个动作。Agent 还需要目标、反馈循环、检查和停止条件。

### 误区二：更多 Agent 一定更快

并行节省等待时间，却增加委派、通信、重复、合并与验证。强依赖任务可能整体更慢。

### 误区三：MCP 接入后就安全可信

MCP 只标准化连接。外部服务器、内容与动作仍需身份、权限、来源和批准治理。

### 误区四：沙箱可以替代验收

沙箱降低环境副作用，不能证明业务方向、内容事实或最终交付正确。

### 误区五：Agent Team 是成熟系统的标志

系统成熟度体现在可控、可观察、可恢复与可评测，不体现在 Agent 数量。

## 岗位迁移

- **研发**：让 Agent 读取仓库、修改最小文件集并运行测试；用 Worktree 隔离并行；发布与高风险命令审批。
- **产品**：独立证据搜集可委派，问题定义与决策统一由主线维护；不要让多个 Agent 各自生成相互矛盾的目标。
- **运营**：素材生成可分支，品牌口径、事实与发布仍统一检查；外部发布必须预览与批准。
- **职能**：可让 Agent 预审材料缺口或整理规则，访问范围按人员身份限制，签发、审批与敏感数据不自动放权。
