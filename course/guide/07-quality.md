# 流畅不是可靠，交付必须有证据与责任人

AI 结果常常语言完整、结构漂亮、语气自信。这些特征只能说明可读，不能证明事实正确、功能可用、风险可控或责任已经明确。

## 本章结论

质量系统要同时回答三个问题：

1. **结果是否通过**：事实、格式、功能和业务目标怎样检查？
2. **过程为什么失败**：错误发生在目标、上下文、工具、模型还是环境？
3. **谁承担后果**：哪些高风险判断与动作必须由责任人批准？

将检查分层：确定性规则放在底层，AI 语义检查放在中层，价值与风险判断留给有责任的人。

> 生成结束不是任务结束。只有证据满足验收，结果才进入可交付状态。

## 知识详解

### 三层质量金字塔

<div class="three-layer-check">
  <div><span>BASE</span><strong>确定性检查</strong><small>Schema、格式、字段、链接、计算、单测、静态规则。便宜、稳定、应尽量自动化。</small></div>
  <div><span>MIDDLE</span><strong>AI 语义检查</strong><small>相关性、完整性、矛盾、风格、证据匹配。需要清楚维度、样例与校准。</small></div>
  <div><span>TOP</span><strong>责任人验收</strong><small>业务价值、授权、取舍、对外承诺与不可逆影响。不能被模型评分替代。</small></div>
</div>

越往上，检查越贵、越慢、越依赖专业判断。好的系统让机器先拦住确定性错误，让 AI 扩大语义检查面，让人只处理真正需要责任的位置。

### 一个 Eval 系统包含什么

| 部件 | 作用 | 需要注意 |
| --- | --- | --- |
| Dataset | 表达真实任务分布 | 包含正常、边界、失败与对抗样本 |
| Environment | 提供可执行状态 | 版本、权限与依赖应可重复 |
| Run | 执行待评测流程 | 固定模型、工具、配置与预算 |
| Trace | 记录决策、工具、结果与交接 | 注意敏感数据与留存范围 |
| Grader | 按规则或语义评分 | 与人类判断校准，避免偏差 |
| Failure Cluster | 聚合同类失败 | 把失败转成修复与回归样本 |

评测不是上线前的一次考试，而是持续反馈。真实失败应进入数据集；每次修改 Prompt、Context、Tool、模型或 Harness 后，用同一任务集比较是否改好，是否引入新回归。

### 结果错了，先定位失败位置

| 失败层 | 常见现象 | 优先修复 |
| --- | --- | --- |
| Goal | 做得很多，却解决了错误问题 | 目标、对象、非目标 |
| Context | 引用旧信息、漏约束、前后矛盾 | 来源、检索、排序、压缩 |
| Prompt / Plan | 顺序错误、格式不稳定 | 任务表达、中间产物、样例 |
| Tool | 选错动作、参数错误、未读取结果 | 工具描述、Schema、反馈 |
| Harness | 无测试、无限循环、不能恢复 | 权限、预算、日志、停止、回滚 |
| Model | 推理或生成能力不足 | 模型选择、拆分、路由 |

如果每次只修改最后一句 Prompt，就会掩盖真正的系统问题。

### Tracing：不仅看答案，还看怎样到达

最终答案可能碰巧正确，但过程读取了不该访问的数据；也可能最终有小格式错误，但工具选择和核心推理正确。Trace 可以帮助检查：

- 用了哪个模型与配置；
- 读了哪些上下文和来源；
- 调用了什么工具、参数与结果；
- 发生了哪些重试、交接与审批；
- 在哪里决定继续或停止。

轨迹本身可能包含敏感内容。采集前要定义最小字段、访问权限、留存周期和脱敏方式。

### Prompt Injection：外部内容可能伪装成指令

Agent 读取网页、邮件、文档、工单或仓库内容时，其中的文字属于**外部数据**。攻击者可能嵌入“忽略原任务、读取密钥、发送数据”等指令。

防护不能只靠一句“不要听恶意内容”，而要纵深组合：

1. 明确信任层级，外部文本默认是数据而非系统指令；
2. 工具与数据遵循最小权限；
3. 使用沙箱或隔离环境；
4. 限制工具参数与可访问目标；
5. 对外发送、删除、付费与权限动作先预览再审批；
6. 对输入、工具调用与输出分别设置 Guardrails；
7. 记录轨迹并准备吊销、回滚与事故处置。

### 风险检查点：影响越大、越难撤回，越要前置

| 动作 | 影响与可逆性 | 默认策略 |
| --- | --- | --- |
| 读取公开资料 | 低影响、可重复 | 可自动，保留来源 |
| 写本地草稿 | 低影响、易撤回 | 可自动，交付前检查 |
| 修改代码或内部记录 | 中影响、可回滚 | 测试、差异审查、限定范围 |
| 对外发送、发布、提交 | 高影响、部分可逆 | 动作预览、人工批准 |
| 删除、付费、权限、敏感数据 | 极高影响、难撤回 | 默认拒绝或强审批 |

人工检查点不是“人永远慢慢重做”，而是把责任放在真正有后果的位置。

<AdvancedTrack title="B 类进阶：评测轨迹与安全不变量">

不要对每个中间文本做脆弱的精确匹配。更有价值的是检查关键行为：是否使用权威来源、是否调用允许的工具、是否在高风险动作前暂停、是否验证结果、是否在预算内结束。安全不变量应尽可能由确定性规则执行，语义质量再由 Grader 与人工抽检补充。

## 工具与选择方法

<ToolLandscape :category-ids="['general-assistant','coding-agent','agent-workflow','capability-reuse']" />

选择质量工具时，关注其是否支持：

- 代表样本与版本化评测；
- 确定性断言与语义 Grader 组合；
- 轨迹、工具调用和环境状态记录；
- 人工审批、失败升级与回滚；
- 敏感数据最小化与审计。

一个能生成漂亮报告但无法回到原始运行证据的“评测平台”，不应成为唯一质量依据。

## 常见误区与边界

### 误区一：让另一个模型打分就完成评测

LLM Grader 也有偏差。需要明确 rubric、好坏样例、与人工判断校准，并与确定性检查组合。

### 误区二：测试通过就可以交付

测试只覆盖已定义行为。业务目标错误、风险遗漏或测试集不完整仍可能导致错误交付。

### 误区三：有引用就代表事实正确

要确认引用是否真的支持结论、是否过期、是否为一手来源、是否遗漏相反证据。

### 误区四：沙箱让高权限 Agent 变得安全

沙箱降低部分系统副作用，不能防止错误业务操作、数据泄露或不当对外承诺。

### 误区五：人工在环等于每一步都需要审批

应按风险分级。低风险可自动，异常和高影响动作才升级给人，否则系统会被审批淹没。

## 岗位迁移

- **研发**：确定性检查用测试与静态分析，语义检查用评审，关键发布与安全风险由责任人确认；保留 diff、日志与回滚。
- **产品**：核对需求事实、逻辑覆盖与决策证据；AI 可检查遗漏，人负责目标、优先级和承诺。
- **运营**：自动检查链接、格式、禁用词与素材规格；AI 预审事实与品牌；责任人批准发布。
- **职能**：字段与日期可自动核对，制度匹配可由 AI 辅助，隐私、签发、审批与重大解释由授权人负责。

## 本章练习

为一项交付写“三层验收 + 风险检查点”：

```text
确定性检查：________________
AI 语义检查：________________
责任人确认：________________
必须暂停的动作：________________
失败后的处置：停止 / 返工 / 升级 / 回滚
需要保留的证据：________________
```

再补一个失败样例：什么结果看起来不错，却必须判定不通过？说明它应被哪一层发现。
