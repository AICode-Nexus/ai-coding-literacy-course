# 先按任务能力选工具，再看产品名称

具体模型、版本、价格、上下文窗口和产品功能变化很快。本页不维护“永远正确”的排名，只提供相对稳定的选型维度。所有动态入口以官方文档为准，核验日期见[来源页](/sources)。

## 能力雷达

| 能力层 | 你真正需要什么 | 常见形态 | 选型时验证 |
| --- | --- | --- | --- |
| 对话与生成 | 表达、总结、分析、结构化输出 | 通用聊天应用 | 中文质量、文件处理、隐私与引用 |
| 深度研究 | 多源检索、证据整理、冲突识别 | Research / Web Agent | 来源可追溯、时效、是否真实打开页面 |
| Coding Agent | 搜索仓库、改代码、跑命令和测试 | IDE Agent / CLI Agent | 工作区权限、diff、测试、回滚、长任务状态 |
| 自动化 Agent | 调用 API、浏览器或业务工具 | Workflow / Agent SDK | 审批、日志、重试、幂等、失败恢复 |
| MCP 连接 | 标准化发现 tools/resources/prompts | Host + MCP Server | 信任边界、权限、维护者、数据范围 |
| Skills | 复用稳定流程、资料、脚本和资产 | Skill 包 / 团队模板 | 触发条件、渐进披露、可测试性 |
| Evals / Tracing | 样本、评分、轨迹与回归 | Eval 平台 / 可观测系统 | 数据集版本、grader 可靠性、成本、隐私 |

## 选型的七个问题

1. 它能否访问完成任务所需的真实材料与环境？
2. 能否限制到最小权限，并在敏感动作前审批？
3. 产出能否留下 diff、日志、引用、测试或轨迹？
4. 失败时能否安全停止、恢复或回滚？
5. 团队能否复用相同的规则、样例、Skill 和 Eval？
6. 数据会被发送到哪里，保留多久，是否符合组织要求？
7. 如果明天换模型，这套工作工件还能不能继续使用？

## 不要用这些信号直接做决定

- 单一排行榜或一次演示；
- 未注明任务分布的“准确率”；
- 未经官方核验的模型版本、窗口或参数表；
- “支持 MCP”但没有权限边界和维护说明；
- “全自动”但没有测试、轨迹、审批与回滚；
- 只比较生成速度，不比较返工、风险和总交付成本。

## 最小工具栈建议

### A 类学员：先完成任务契约

一个可靠的通用 AI + 真实材料 + 任务卡 + 验收清单就足够。不要因为工具焦虑跳过目标和材料。

### B 类学员：补齐 Coding Agent 的 Harness

在 IDE/CLI Agent 之外，重点检查项目规则、最小权限、测试命令、Git 状态、浏览器验证、停止条件和交付证据。

### 团队实践：再考虑连接与复用

只有重复工作已经稳定后，再封装 Skill；只有真实工具需要跨 Host 复用时，再接 MCP；只有子任务可独立、收益大于协调成本时，再增加 Subagent 或 Team。

::: tip 判断工具是否真正适合
用你的 5–10 个真实任务样本做小型 Eval，比浏览更多功能列表更可靠。记录成功率、人工介入、返工、总耗时和高风险失败，而不只看生成速度。
:::

## 动态知识的维护规则

- 稳定方法进入正文；
- 产品能力只链接官方入口；
- 时效结论写核验日期；
- 内部资料中的版本与排名不直接复制；
- 每次正式授课前检查“dynamic”来源；
- 过时内容直接删除，不为了保留历史而干扰主线。
